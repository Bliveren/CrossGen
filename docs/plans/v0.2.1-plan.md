# Image2Tools v0.2.1 实现计划（优化版）

> 状态：草案，待实现。基于 v0.2.0 发布后收到的三条用户反馈。
> 本文件由 v0.2.0 上线后的代码快照核对过行号与现状（详见每节"现状核对"）。
> 实现时仍以符号/代码搜索定位为准，行号仅供参考——任何编辑前先 Read 确认。

## 背景

用户反馈三个问题需要在 v0.2.1 中修复：

1. **服务商下拉框多余** — 用户配置 base URL 后应自动识别服务商并进行模型适配分类。
2. **参考图编辑失败** — 聚合器不支持 streaming edit，返回错误：
   `OpenAI API 请求失败：Error reading stream: provider returned non-SSE response for streaming request`。
3. **生成/编辑模式区分不清** — 三个 tab（生成/编辑/局部重绘）让用户困惑，简化为"文生图"和"图生图"两个 tab。

## 关键前置事实（实现前必读）

这些是核对真实代码后得到的、与原始计划不同或原始计划未覆盖的要点：

- `mode` 字段**不在** `OpenAIImageParams` 上，而在 `GenerationJob` 上（`OpenAIImageJob = GenerationJob & { params: OpenAIImageParams }`，`src/main/services/openaiImageAdapter.ts:64`）。因此 Change 2 的判断必须读 `job.mode`，不能读 `params.mode`。
- 现有"`n > 1` 禁用 streaming"的逻辑在 `normalizeOpenAIRequestParams`（`openaiImageAdapter.ts:89–96`），**不在** `normalizeOpenAIJobParams`。但 `normalizeOpenAIRequestParams` 只接收 `params`、拿不到 `mode`，所以 Change 2 的 mode 判断只能放在 `normalizeOpenAIJobParams`（230–233，能拿到 `job.mode`）。两处分工要分清。
- streaming 字段命名：内部 params 用 `stream`（boolean）与 `partialImages`（number 0–3）；序列化到请求体时才是 `partial_images`。改的是内部字段。
- 渲染层 `saveConfig` **当前已经传 `kind`**（`App.tsx:996`），Change 1 要把它改成不传。
- draft 当前保存 `mode`，**不保存 `kind`**（`App.tsx:827–836`）。Change 3 要新增 tabMode 持久化。
- 模式 tab 当前由 `modeLabels` 动态生成（`App.tsx:1987`），且 Gemini guided-region 会把 `inpaint` 的文案替换成 `guidedRegionMode`（`modeLabelsForParams`，464–468）。Change 3 重构 tab 时**必须保留 guided-region 文案分支**，否则 Gemini 局部引导编辑的提示会回退成"局部重绘/exact mask"措辞，违反已通过的发布证据约束。
- 版本号改动会级联触发 `verify:release-evidence`（要求 `evidence.json.releaseVersion === package.json.version`，`scripts/verify-release-evidence.mjs:352`）和 `package-config.test.ts:7`（硬断言 `"0.2.0"`）。详见 Change 4。

## Change 2: 编辑请求禁用 streaming

### 问题根因

聚合器/代理对 `/images/edits` 端点不支持 SSE 流式响应。当 `stream: true` + `Accept: text/event-stream`（`openaiImageAdapter.ts:335`）发出时，聚合器返回非 SSE 响应，在流读取阶段抛出 `provider returned non-SSE response for streaming request`。

注意这是**产品取舍**而非纯 bug：官方 OpenAI 端点的 edits 是支持 streaming 的，这里选择对所有后端统一关闭 edit/inpaint 的 streaming，以换取聚合器兼容性。编辑场景 partial preview 价值低（用户已有源图），取舍可接受。

### 修改方案

在 `normalizeOpenAIJobParams()` 中，当 `job.mode !== "generate"` 时强制关闭 streaming。放这里是因为它能拿到 `job.mode`；`normalizeOpenAIRequestParams` 拿不到 mode，保持不动。

### 修改文件

**`src/main/services/openaiImageAdapter.ts`（约 230–233，实现前 Read 确认）**

将：
```typescript
function normalizeOpenAIJobParams(job: OpenAIImageJob): OpenAIImageJob {
  const params = normalizeOpenAIRequestParams(job.params);
  return params === job.params ? job : { ...job, params };
}
```

改为：
```typescript
function normalizeOpenAIJobParams(job: OpenAIImageJob): OpenAIImageJob {
  let params = normalizeOpenAIRequestParams(job.params);
  // 聚合器普遍不支持 /images/edits 的 SSE 流式响应；编辑/局部重绘统一走非流式。
  if (job.mode !== "generate" && params.stream) {
    params = { ...params, stream: false, partialImages: 0 };
  }
  return params === job.params ? job : { ...job, params };
}
```

### 设计理由

- 不做 retry（重试增加延迟和复杂性）。
- 与现有 `n > 1` 禁用 streaming 的取舍一致（同样是收窄行为）。
- 收窄行为，不影响文生图的流式预览。

### 测试

- `src/main/services/openaiImage.test.ts`：新增用例——`mode: "edit"` / `"inpaint"` + `stream: true` 经 `normalizeOpenAIJobParams` 后 `stream === false` 且 `partialImages === 0`；`mode: "generate"` + `stream: true` 保持 `stream === true`。
- `src/main/services/verify-mock-api`（mock OpenAI verifier）：确认 edit 请求体不再带 `stream`/`partial_images`。

## Change 1: 移除服务商下拉，自动检测

### 现有能力（现状核对）

- `modelDiscovery.ts` 已有 `inferredProviderKind` 机制（`discoverModelsAcrossProviders`，47–82）：选定 provider 没发现模型、但恰好另一个 provider 发现了，则推断出正确 provider（仅当只有一个其他 provider 成功时）。
- `refreshModelDiscovery()`（`main.ts:565–598`）已用 `discovery.inferredProviderKind ?? config.kind` 覆盖 `config.kind`。
- `selectActiveLaunchForDiscovery()`（`main.ts:600–643`）已按推断 provider 自动选 launch family。
- `discoveryProviderOrder("custom")` 当前为 `["custom", "gemini"]`（`modelDiscovery.ts:89–93`）——缺 `openai`，未指定 provider 时不会尝试 OpenAI 协议。

### 修改文件

#### 1. `src/main/main.ts` — `handleSaveConfig()` 增强（约 502–536）

- UI 不再提交 `kind`，**默认保留已存 kind**（`buildProviderConfigForSave` 在 `input.kind === undefined` 时保持 `current.kind`）。不要无条件用 base URL 猜 kind：猜出的 kind 一旦与已存 kind 不同，会触发 `buildProviderConfigForSave` 的 "provider changed" 分支，静默清除已保存的 Key 并把旧 provider 的 launch 带到新 provider 上（见"代码评审修正"）。
- provider 重新分类交给 model discovery 的 `inferredProviderKind` 非破坏式处理。
- 触发 discovery 的条件放宽：当前仅在 `input.apiKey` 非空时触发；改为**已有 key 的情况下改 baseURL 也触发**（换聚合器地址需重新分类）。无 key 时不触发（discovery 需要鉴权）。

#### 2. `src/renderer/App.tsx` — 移除 provider dropdown

- 删除 provider `<select>` 元素与 `changeProvider()` 函数。
- `saveConfig()` 中**移除 `kind: providerKind` 字段**，不再由 UI 提交 kind；`launchModel`/`selectLaunchModel` 的 `saveConfig` 调用同样移除 `kind`。
- 保留 `providerKind` state 作为**只读显示**（由 snapshot 同步设置），展示"已识别为 OpenAI / Gemini / Custom"。

#### 3. `src/main/services/modelDiscovery.ts` — 维持 custom 探测顺序

`discoveryProviderOrder("custom")` **保持** `["custom", "gemini"]`。custom 本身就是 OpenAI 兼容协议，再加 `openai` 会对同一 `/models` 端点探测两次、把通用模型按两个 tag（`openai:` 与 `custom:`）各列一遍（`uniqueModels` 按 `providerKind:id` 去重），导致 General 下拉重复、发现计数翻倍（见"代码评审修正"）。仅导出该函数供测试。

### 向后兼容

- `ProviderConfigInput.kind` 已是 optional——不传时 `buildProviderConfigForSave` 保持现有 kind。
- 已保存配置中的 `kind` 不丢失，只是不再由 UI 编辑。
- 自定义聚合器域名 → 依赖 model discovery 跨协议探测（custom→gemini）与 `inferredProviderKind` 推断重新分类。
- **不**用 URL 启发式翻转 kind，避免静默清 Key / 错配 launch。

### 测试

- `src/shared/validation.test.ts`：新增 `inferProviderKindFromURL` 用例——官方 OpenAI URL → `"openai"`；官方 Gemini URL → `"gemini"`；聚合器/未知域名 → `undefined`；尾斜杠/大小写归一化。
- `src/main/services/modelDiscovery.test.ts`：新增用例——`discoveryProviderOrder("custom")` 返回包含 `openai` 的顺序；已有"推断 provider"的用例保持通过。
- `src/main/main.test.ts`（若存在 handleSaveConfig 覆盖）：改 baseURL（已有 key、未传 kind）会触发一次 discovery；无 key 时不触发。

## Change 3: 简化为"文生图/图生图"两个 Tab

### 设计方案

将三个 tab（生成/编辑/局部重绘）合并为两个：
- **文生图**（text2img）— 仅提示词，内部 `generate` → `/images/generations`。
- **图生图**（img2img）— 参考图 + 可选蒙版，内部 `edit`（无 mask）或 `inpaint`（有 mask）→ `/images/edits`。

蒙版从"独立 tab"降为图生图下的可选输入。

### 内部路由不变

API 路由仍由内部 `WorkMode`（generate/edit/inpaint）控制；UI 层新增 `TabMode`：
- `text2img` → `generate`
- `img2img` + 无 mask → `edit`
- `img2img` + 有 mask → `inpaint`

### 必须保留的现有行为（关键）

- **Gemini guided-region 文案分支**：`modeLabelsForParams`（`App.tsx:464–468`）在 Gemini guided-region 能力下把 `inpaint` 文案替换为 `guidedRegionMode`。重构 tab 后，img2img + mask 在 Gemini 下的标题/动作/描述仍须走 guided-region 文案（"区域引导"），不得回退成"局部重绘/exact mask"。`maskDescription`（1520）的 guided-region 分支同样保留。
- **General 模式**：`isGeneralMode` 时当前不显示 mode tabs，而是显示 general 状态条（1980–1984）。两 tab 改造只作用于非 general 分支；general 的 requestMode 推断（`generalAllowsReferences && inputAssets.length > 0 ? "edit" : "generate"`，1251/1282）保持不变。

### 修改文件

#### 1. `src/renderer/App.tsx`

**新增状态：**
```typescript
type TabMode = "text2img" | "img2img";
const [tabMode, setTabMode] = useState<TabMode>("text2img");
```

**计算实际 requestMode（替换约 1282 的 `requestMode` 计算）：**
```typescript
const requestMode: WorkMode = generalParams
  ? (generalAllowsReferences && inputAssets.length > 0 ? "edit" : "generate")
  : tabMode === "text2img"
    ? "generate"
    : (maskAsset || maskDataUrl) ? "inpaint" : "edit";
```

**Tab UI 替换（约 1986–2004，非 general 分支）：** 用两个固定按钮替换 `modeLabels` 遍历。文案取自新增 i18n（见下），img2img 在 Gemini guided-region 下的提示词仍由 `modeLabels`/guided-region 分支决定。

**参考图/蒙版可见性：**
- 参考图工具：非 general 且 `tabMode === "img2img"` 时显示。
- 蒙版区域（当前 `mode === "inpaint" && !isGeneralMode`，约 2176）改为 `tabMode === "img2img" && !isGeneralMode`，并在标题旁加"（可选）"标签。

**Run 按钮文案（约 2110，当前 `modeLabels[mode].action`）：**
- text2img → 生成
- img2img 无 mask → 编辑
- img2img 有 mask → 局部重绘（Gemini guided-region 下 → 区域引导）
- 即：按 `requestMode` 取 `modeLabels[requestMode].action`，复用现有 guided-region 文案逻辑。

**结果/历史标题**（2013、2303 用 `modeLabels[job.mode].title`）：保持读 `job.mode`，无需改（历史任务仍存内部 WorkMode）。

**自动行为：**
- 切到 text2img 不清除已选参考图（保留状态），runJob 时 inputPaths = []。
- 切到 img2img 不自动加图（用户手动）。

**Draft save/restore（约 824–843 / 909–926）：**
- 保存 `tabMode` 到 draft（新增字段）。
- 恢复优先用 draft.tabMode；缺省时由 `mode` 推导：`generate → text2img`，`edit`/`inpaint → img2img`（兼容 v0.2.0 旧 draft，旧 draft 无 tabMode）。

**modeError 简化：**
- text2img：无需校验参考图/蒙版。
- img2img：参考图为空时报错要求添加；蒙版校验仅在蒙版存在时检查格式（不再因"inpaint 缺 mask"报错——img2img 无 mask 是合法的 edit）。

#### 2. `src/renderer/i18n.ts`（EN 约 335–340 / ZH 约 530–535）

在现有 `modes` / `guidedRegionMode` 旁新增 tab 文案（保留 `modes`，因为 Run 按钮、结果标题、历史仍按内部 WorkMode 取文案）：

i18n shape 测试需同步：两种语言的 `tabs`/`maskOptional` 键齐全。

#### 3. `src/renderer/styles.css`

- 复用现有 `.mode-tabs` / `.mode-tab`（两个 tab 更宽敞，无需新结构）。
- 蒙版区域新增 `.mask-optional` 标签样式。

### 测试

- `src/shared/validation.test.ts`：job-run 校验调整——img2img 无 mask（即 `edit` 模式）合法；现有"非 inpaint 携带 mask / inpaint 缺 mask"等 IPC 校验仍以内部 WorkMode 为准（后端契约不变，UI 不会再产出"inpaint 无 mask"）。确认这批断言不被破坏。
- 渲染层（若有组件测试）：tabMode 切换 → requestMode 映射正确；旧 draft（仅 mode）恢复落到正确 tab。
- i18n 测试：新键齐全。

## 代码评审修正（实现后高强度自审发现）

实现完成后对整个 diff 做了 high-effort code review，发现并修复了三处由本次改动引入的回归：

1. **freehand 蒙版绘制失效（阻断级）**：`startPaint` 原 gate 为 `requestMode === "inpaint"`，但 `requestMode` 只有在已有蒙版时才会变成 `inpaint`，而绘制本身才是创建蒙版的途径——形成死循环，img2img 下无法手绘蒙版。改为 gate `tabMode === "img2img" && sourcePreview && !isGeneralMode`（即画布显示的条件），让绘制能产生蒙版、`requestMode` 再据此变为 `inpaint`。
2. **改 base URL 静默清 Key / 错配 launch**：原实现用 `inferProviderKindFromURL` 在 `input.kind` 缺省时翻转 kind，导致 Gemini 用户把 base URL 改成 `api.openai.com` 时 `providerChanged` 触发，清空已存 Key、并把 Gemini 的 launch/model 带到 openai 上。移除该启发式翻转，默认保留已存 kind，重新分类只交给 discovery 的 `inferredProviderKind`。
3. **custom 发现重复模型**：`discoveryProviderOrder("custom")` 改成含 `openai` 会对同一 `/models` 端点探测两次、通用模型按两个 tag 各列一遍。回退为 `["custom", "gemini"]`。

`inferProviderKindFromURL` 因此不再被使用，已从 `validation.ts` 删除（连同其测试）。
