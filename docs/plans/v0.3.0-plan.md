本次审核结论：

- 原计划方向正确，但需要从“功能想法清单”收敛为“可拆 PR、可验收、可回滚”的执行计划。
- `v0.3.0` 应聚焦多 API 管理和提示词生产力，不把重试、队列、CLI、智能降级等基础设施扩展塞入同一版本。
- `STATE_VERSION = 3` 已经随 `v0.2.4` multi-provider 基线使用；新增模板和 Gallery 字段按 v3 可选扩展处理，除非出现破坏性 schema 变化，否则不再 bump。
- 多 API 切换是最高风险路径，必须保护用户 prompt、参考图、Key、模型发现结果和当前草稿，不能用清空状态换取实现简单。
- Gallery 是最高安全风险路径，必须采用受管目录复制和路径归属校验，不能让 renderer 通过绝对路径读取任意本地文件。
- Gallery 在 `v0.3.0` 中增加“文件夹式素材管理”：用户可创建自定义文件夹，并把图片素材收纳到不同文件夹；实现上先采用逻辑文件夹，不把用户可见文件夹映射为真实磁盘目录，避免引入路径越权和跨目录迁移风险。
- chips 输入法必须保持 provider adapter 无感知，最终仍序列化为纯文本 prompt 和 input assets。

执行原则：

- 每个阶段独立分支、独立 PR、独立验收。
- 阶段顺序不得倒置：先多 API UI，再模板，再 Gallery，最后 chips。
- 每个阶段合并前必须完成对应 checklist 和验收命令。
- release 收尾前不得提交 `.claude/`、临时计划草稿、真实 API Key、证书、签名密码或本机路径敏感产物。

## 1. 版本目标

`v0.3.0` 只做两条主线：

1. 多 API 管理：用户可以保存多个 API 接入配置，在 OpenAI / Gemini / Custom 之间切换，不需要反复输入 Key、Base URL 和模型发现结果。
2. 提示词生产力与素材管理：提供提示词模板、参考图 Gallery、Gallery 文件夹管理、chips 输入法，让提示词和参考素材可以复用、分类、组合、分享。

成功标准：

- 用户能保存至少两个 API 接入，并可靠切换运行。
- 用户能创建模板、导入参考图到 Gallery，并在 prompt 中组合模板 / Gallery 引用 / 色值 chip。
- 用户能创建 Gallery 文件夹，把图片素材移动到不同文件夹，并按文件夹、标签、搜索结果快速定位素材。
- 旧 `v0.2.x` 状态文件可自动迁移，不丢历史、Key、草稿和已有输出。
- 下游 `runJob` provider adapter 契约保持稳定：最终仍提交纯字符串 `prompt` + `inputAssets` + `params`。

## 3. 范围边界

本版本包含：

- 多 API 接入选择器、添加、删除、切换、连接状态展示。
- 提示词模板本地库。
- 参考图 Gallery 本地库与文件夹式素材分类管理。

- 单任务多图的队列化体验。
- CLI。

## 4. 状态版本策略

当前 `STATE_VERSION = 3` 已随 `v0.2.4` 用于 multi-provider。`v0.3.0` 不再因为新增可选字段而 bump 到 4，除非出现破坏性 schema 变化。

`v0.3.0` 新增字段均按 v3 可选扩展处理：

- `promptTemplates?: PromptTemplate[]`
- `galleryAssets?: GalleryAsset[]`
- `galleryFolders?: GalleryFolder[]`
- 可选的 draft prompt token 字段（仅在 chips 阶段需要时添加）

迁移规则：

- 旧 `config` 单配置状态继续迁移为 `providers[0]`。
- 缺少 `promptTemplates` 时补 `[]`。
- 缺少 `galleryAssets` 时补 `[]`。
- 缺少 `galleryFolders` 时补 `[]`。
- `GalleryAsset.folderId` 指向不存在文件夹时置为 `null`，不能导致素材丢失或应用启动失败。
- 任一数组中畸形记录要过滤或规范化，不能导致应用启动失败。
- 每次状态写入继续沿用现有原子写 + 备份机制。

## Phase 3：参考图 Gallery

目标：

- 用户能把本地参考图或历史结果加入 Gallery。
- Gallery 中的图可以跨任务复用。
- Gallery 提供文件夹式素材管理：用户可创建、重命名、删除文件夹，并将图片素材收纳到指定文件夹。
- 文件夹是逻辑分类，不对应真实磁盘目录；所有图片文件仍统一复制到 `userData/gallery` 受管目录。
- 所有 Gallery 预览仍受 managed asset 协议保护，不暴露任意本地路径。
- 为 Phase 4 的 `@gallery` chip 提供稳定数据源。

主要文件：

- `src/shared/types.ts`
- `src/main/services/stateMigration.ts`
- `src/main/services/assetOwnership.ts`
- `src/main/main.ts`
- `src/preload/preload.ts`
- `src/renderer/App.tsx`
- `src/renderer/i18n.ts`
- `src/renderer/styles.css`
- `src/main/services/assetOwnership.test.ts`
- `src/renderer/App.smoke.test.tsx`

数据模型：

```ts
export interface GalleryFolder {
  id: string;
  name: string;
  color?: string;
  createdAt: string;
  updatedAt: string;
}

export interface GalleryAsset {
  id: string;
  fileName: string;        // userData/gallery 下的相对文件名
  originalName: string;
  mimeType: string;
  sizeBytes: number;
  width?: number;
  height?: number;
  tags: string[];
  folderId?: string | null;
  source: "import" | "result";
  createdAt: string;
  updatedAt: string;
}

export interface GalleryAssetPatch {
  tags?: string[];
  folderId?: string | null;
}

export interface GalleryFolderInput {
  name: string;
  color?: string;
}
```

文件夹策略：

- `folderId: null` 或缺失表示“未分类”。
- `v0.3.0` 只支持一层自定义文件夹，不支持嵌套文件夹。
- 标签用于多维检索，文件夹用于主归档；两者可以同时存在。
- 删除文件夹不删除图片文件，默认把该文件夹下素材移动到“未分类”。
- 文件夹名称允许重复前应先评估 UX；建议 `normalizeGalleryFolderName()` 后不允许同名。
- 文件夹颜色仅用于列表识别，不参与业务逻辑。

Phase 3 不做：

- 嵌套文件夹、文件夹拖拽排序、智能分类。
- 将 Gallery 文件夹同步为真实磁盘目录。
- 监听用户磁盘目录变化或自动导入。
- 跨设备同步、云端素材库、共享文件夹。
- 批量移动/批量删除作为必须项；若实现批量操作，必须补 smoke test 和二次确认。

State checklist：

## 6. 风险与控制

| 风险 | 等级 | 控制措施 |
| --- | --- | --- |
| 多 API 切换导致用户输入丢失 | 高 | 切换前保存草稿；不静默清空 prompt/input；仅显示兼容性提示 |
| v3 状态字段继续扩展导致迁移混乱 | 中 | 明确新字段为 v3 可选扩展；新增 normalize 测试 |
| Gallery 暴露任意本地路径 | 高 | 只复制到 `userData/gallery`；协议只读受管目录；symlink 越界测试 |
| Gallery 文件夹删除导致素材误删 | 高 | 文件夹是逻辑分类；删除文件夹只把素材移到未分类；删除前明确二次确认 |
| 文件夹 / 标签 / 搜索组合筛选状态混乱 | 中 | 单一 selector 管理筛选状态；smoke test 覆盖组合筛选和计数更新 |
| chips 输入法影响纯文本体验 | 中 | 独立序列化器；纯文本 smoke test 必须保持 |
| `App.tsx` 继续膨胀 | 中 | Phase 4 优先抽 `PromptComposer` 和 `promptTokens` |
| 范围膨胀拖延 v0.3.0 | 高 | retry、队列、CLI、智能降级全部移出 v0.3.0 |
| macOS notarization 卡住发布 | 中 | release evidence 如实记录；发布前分别确认 `macos-signed` 与 `macos-notarized` gate 状态 |

## 7. 完成定义

`v0.3.0` 视为完成必须同时满足：

- Phase 1-4 checklist 全部完成。
- 所有自动化验证通过。
- 串联手动验收通过。
- README / RELEASE_NOTES / release evidence / update manifest 已更新。
- GitHub release 资产与 manifest 中 hash/size 完全一致。
- 未提交任何本地计划草稿、`.claude/`、真实 API 产物、证书或凭据。
