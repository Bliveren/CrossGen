# CrossGen v0.3.1 CLI + MCP Plan

> Planning date: 2026-07-06
> Target version: `v0.3.1`
> Theme: CLI and local MCP access for CrossGen Gallery, provider status, and controlled generation.
> Boundary: `v0.3.0` remains the release for multi API management, prompt productivity, Gallery folders, and prompt chips. CLI and MCP are the main `v0.3.1` plan.

## 0. Decision

`v0.3.1` should ship CLI + MCP as an infrastructure-focused release, but not as a single high-risk jump. The order is:

1. Freeze runtime and key reuse behavior.
2. Extract a non-Electron core.
3. Ship read-only CLI and read-only MCP.
4. Add Gallery write operations behind explicit write mode.
5. Add paid generation only after key reuse and cost gating are verified.

Hard corrections from the initial proposal:

- Do not assume `Electron-as-Node` can decrypt existing `safeStorage` keys. A local probe showed `ELECTRON_RUN_AS_NODE=1 ./node_modules/.bin/electron -e ...` does not expose `safeStorage`.
- Prefer a normal Electron runtime command mode, for example `CrossGen --cli ...` and `CrossGen --mcp`, so packaged builds can still use `app.getPath()` and `safeStorage`.
- MCP defaults to `readonly`, not `write`.
- Data directory detection must support the legacy `Image2Tools` userData directory and state-level `storage.historyDir` / `storage.galleryDir` overrides.
- CLI/MCP writes must use a cross-process state lock or route through the same Electron process.
- Gallery folders are real disk directories in the current implementation, so CLI/MCP must reuse the same managed-path and disk reconciliation rules.

## 1. Product Goal

Let local coding agents such as Claude Code and Cursor use CrossGen safely:

- Inspect provider status, discovered models, Gallery assets, folder trees, and asset metadata.
- Use a stable CLI JSON contract for scripts and agent workflows.
- Use a local stdio MCP server with schema-defined tools.
- Import, move, rename, and remove Gallery assets only after explicit write mode is enabled.
- Generate or edit images only after explicit generate mode and cost confirmation.

Non-goals:

- No cloud sync, accounts, collaboration, or hosted service.
- No HTTP server or local network port.
- No bypass of desktop app safety rules.
- No new provider adapter overhaul in `v0.3.1`.
- No promise that pure Node can decrypt Electron `safeStorage` secrets.

## 2. Current Baseline

Known reusable pieces:

- Electron + React + TypeScript + Vite desktop app.
- Package name is `crossgen`; product name is `CrossGen`.
- State file is still `image2tools-state.v1.json`.
- Desktop userData preserves legacy `Image2Tools`.
- `AppStateFile` already contains providers, active provider id, prompt templates, Gallery folders/assets, storage settings, and draft state.
- Provider adapters are close to reusable core code:
  - `src/main/services/imageProviderAdapter.ts`
  - `src/main/services/imageProviderAdapters.ts`
  - `src/main/services/openaiImageAdapter.ts`
  - `src/main/services/geminiImageAdapter.ts`
  - `src/main/services/generalImageAdapter.ts`
  - `src/main/services/modelDiscovery.ts`
- Gallery path safety and disk sync already exist:
  - `resolveManagedFileName`
  - `assertManagedRegularFile`
  - `assertManagedRegularFileInRoots`
  - `scanGalleryDisk`
  - `reconcileGalleryDiskState`
  - `reconcileGalleryDiskChanges`
  - `startGalleryDiskWatchers`

Current coupling to unwind:

- `src/main/main.ts` owns Electron lifecycle, IPC, state read/write, key encryption, Gallery mutations, job orchestration, dialog, and shell integration.
- `safeStorage` encryption/decryption still lives in `main.ts`.
- `galleryOperationQueue` is process-local and does not protect CLI/MCP processes.
- `sendJobEvent` and `sendGalleryEvent` depend on `BrowserWindow`.
- Build config does not include `src/cli` or `src/mcp` yet.

## 3. Target Architecture

```text
src/core/
  dataDirs.ts       platform and explicit data-dir resolution
  stateStore.ts     normalized state read/write + lock
  keyring.ts        injected key resolver contract
  gallery.ts        Gallery scan, CRUD, disk reconciliation
  providers.ts      active provider and model selection
  generation.ts     job creation and adapter orchestration

src/main/
  Electron lifecycle, UI IPC, safeStorage keyring, shell/dialog adapters

src/cli/
  CLI parser, JSON output contract, stderr logs

src/mcp/
  stdio MCP server, tool schemas, mode-based registration
```

Core rules:

- `src/core` must not import `electron`.
- All state writes go through `stateStore.withStateLock`.
- Gallery file mutations and state writes happen in one critical section.
- Electron main, CLI, and MCP share the same core APIs.
- MCP should call core directly instead of spawning CLI.

## 4. Key Reuse

Supported paths:

| Runtime | `safeStorage` | `localFallback` | env key |
| --- | --- | --- | --- |
| Desktop UI | yes | yes | optional |
| Normal Electron command mode | target, must verify | yes | yes |
| Pure Node CLI/MCP | no | yes | yes |

Preferred runtime:

```bash
CrossGen --cli config status --json
CrossGen --mcp
```

Requirements:

- App startup detects `--cli` / `--mcp` before creating a window.
- Command mode can access `app.getPath()` and `safeStorage`.
- Command mode exits explicitly after command completion.
- Packaged macOS, Windows, and Linux builds must be tested.

Env key precedence:

1. Provider-specific env key:
   - `CROSSGEN_OPENAI_API_KEY`
   - `CROSSGEN_GEMINI_API_KEY`
   - `CROSSGEN_CUSTOM_API_KEY`
2. `CROSSGEN_API_KEY`
3. Saved provider key
4. Error

Generation must not require users to resave keys in a less secure mode. If packaged `safeStorage` reuse cannot be verified, generation can ship as env-key-only or move to `v0.3.2`.

## 5. Data Directories

Resolution priority:

1. `--data-dir <path>`
2. `CROSSGEN_DATA_DIR`
3. Electron command mode `app.getPath("userData")`
4. Legacy platform paths:
   - macOS: `~/Library/Application Support/Image2Tools`
   - Windows: `%APPDATA%/Image2Tools`
   - Linux: `~/.config/Image2Tools`, with compatibility for an existing lowercase directory
5. Error if `image2tools-state.v1.json` is not found

After reading state, core must respect:

- `state.storage.historyDir`
- `state.storage.galleryDir`

Do not assume Gallery lives under userData.

## 6. CLI Contract

Global options:

- `--json`
- `--data-dir <path>`
- `--provider <id-or-name>`
- `--mode readonly|write|generate`
- `--yes`

JSON success:

```json
{
  "ok": true,
  "data": {}
}
```

JSON error:

```json
{
  "ok": false,
  "error": {
    "code": "CONFIG_NOT_FOUND",
    "message": "..."
  }
}
```

Rules:

- `--json` stdout contains exactly one JSON object.
- Logs go to stderr.
- Exit code matches `ok`.
- Errors are secret-redacted.

Commands:

```bash
crossgen config status [--json]
crossgen provider list [--json]
crossgen models list [--provider <id>] [--refresh] [--json]
crossgen doctor [--json]

crossgen folder tree [--json]
crossgen gallery list [--folder <id>] [--tag <tag>] [--query <text>] [--json]
crossgen asset inspect <id> [--json]
crossgen asset path <id> [--json]

crossgen folder create <name> [--parent <id>] [--json]
crossgen folder rename <id> <name> [--json]
crossgen folder move <id> --parent <id|null> [--json]
crossgen folder delete <id> --yes [--json]
crossgen asset import <path...> [--folder <id>] [--json]
crossgen asset move <id> --folder <id|null> [--json]
crossgen asset remove <id> --yes [--json]
crossgen asset open <id>

crossgen generate --prompt <text> [--model <id>] [--folder <id>] --yes [--json]
crossgen edit --prompt <text> --input <path-or-asset-id> [--mask <path>] [--folder <id>] --yes [--json]
```

`models list` uses cached `discoveredModels` by default. `--refresh` can call the provider API, requires a resolvable key, and does not create paid image outputs.

## 7. MCP Contract

Transport:

- stdio only.
- No HTTP server.
- No port binding.

Mode:

- `CROSSGEN_MCP_MODE=readonly|write|generate`
- Default is `readonly`.

Tool registration:

| Mode | Tools |
| --- | --- |
| `readonly` | `config_status`, `provider_list`, `models_list`, `folder_tree`, `gallery_list`, `asset_inspect` |
| `write` | readonly tools plus `folder_create`, `folder_rename`, `folder_move`, `folder_delete`, `asset_import`, `asset_move`, `asset_remove` |
| `generate` | write tools plus `generate_image`, `edit_image` |

Paid tool rules:

- `generate_image` and `edit_image` are only registered in `generate` mode.
- Tool input must include `confirm: true`.
- Results include provider, model, job id, output asset ids, and a cost reminder.
- Results must not include raw API keys, bearer tokens, private base URLs, or full local paths unless the tool is explicitly path-oriented.

Example MCP config:

```jsonc
{
  "mcpServers": {
    "crossgen": {
      "command": "/Applications/CrossGen.app/Contents/MacOS/CrossGen",
      "args": ["--mcp"],
      "env": {
        "CROSSGEN_MCP_MODE": "readonly"
      }
    }
  }
}
```

## 8. Phases

### Phase 0: v0.3.0 Release Gate And Runtime Spike

Start only after `v0.3.0` is released or explicitly paused.

Tasks:

- Verify packaged `CrossGen --cli config status --json`.
- Verify whether normal Electron command mode can decrypt saved `safeStorage` keys.
- Freeze CLI JSON and MCP tool contracts.

Acceptance:

- macOS dev and packaged command mode both run.
- The safeStorage result is documented.
- If safeStorage fails, generation scope is downgraded to env-key-only or deferred.

### Phase 1: Core Extraction And State Lock

Tasks:

- Move data-dir resolution, state read/write, Gallery reconciliation, Gallery CRUD, provider selection, and job orchestration into `src/core`.
- Add cross-process lock.
- Keep desktop behavior unchanged.

Acceptance:

- `pnpm typecheck`
- `pnpm test`
- `pnpm build`
- Existing Gallery and state migration tests pass.
- No `STATE_VERSION` bump unless a breaking schema change is introduced.

### Phase 2: Read-Only CLI And MCP

Tasks:

- Implement read-only CLI commands.
- Implement readonly MCP server.
- Add command contract tests and MCP tool registration tests.

Acceptance:

- CLI stdout JSON snapshots pass.
- MCP default mode registers only read-only tools.
- No state write occurs.
- No secret appears in stdout, stderr, or MCP results.

### Phase 3: Gallery Write Operations

Tasks:

- Implement folder and asset mutation commands.
- Implement write-mode MCP tools.
- Require `--yes` or `confirm: true` for destructive operations.

Acceptance:

- Path traversal, symlink escape, invalid filename, and Windows reserved name tests pass.
- CLI import is visible to the running desktop app after watcher sync.
- Concurrent desktop + CLI mutation does not lose state.

### Phase 4: Generation And Editing

Tasks:

- Implement generation commands behind `--mode generate` and `--yes`.
- Implement `generate` MCP mode.
- Reuse active provider or `--provider`.
- Save outputs to history and optionally Gallery.

Acceptance:

- Mock OpenAI and Gemini generation pass.
- MCP default mode still does not register paid tools.
- `safeStorage` key reuse works in verified packaged runtimes, or generation clearly requires env key.

### Phase 5: Docs, Packaging, Release

Tasks:

- Include CLI/MCP entries in build output and electron-builder files.
- Add Chinese and English user docs.
- Add MCP config examples.
- Add release evidence for v0.3.1.

Acceptance:

- `pnpm build`
- `pnpm verify:mock-api`
- `pnpm verify:mock-gemini-api`
- `pnpm verify:mock-model-discovery`
- Packaged CLI smoke
- MCP readonly smoke
- Release evidence updated for actual v0.3.1 assets

## 9. Security Checklist

- [ ] `src/core` has no Electron imports.
- [ ] `safeStorage` is used only through injected Electron keyring.
- [ ] Pure Node does not attempt to decrypt `safeStorage`.
- [ ] All state writes use cross-process lock.
- [ ] Gallery paths use managed-path validation.
- [ ] CLI/MCP default to readonly.
- [ ] Write and generate modes require explicit opt-in.
- [ ] Paid operations require `--yes` or `confirm: true`.
- [ ] API keys are never echoed.
- [ ] Provider errors are secret-redacted.
- [ ] No HTTP server or local port is opened.
- [ ] Folder delete follows desktop semantics and does not delete assets unless separately designed.

## 10. Release Boundary

Required for `v0.3.1`:

- Core extraction.
- Cross-process state lock.
- Read-only CLI.
- Read-only MCP.
- Gallery write operations.
- Security and contract tests.
- User documentation.

Optional for `v0.3.1`:

- Generate and edit commands/tools.

If packaged `safeStorage` reuse is not verified, paid generation should ship as env-key-only or be deferred to `v0.3.2`. Do not ask users to weaken key storage to meet the release date.
